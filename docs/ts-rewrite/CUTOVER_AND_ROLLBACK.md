# 上线切换与回退说明

## 目标
- 降低 TS 重写上线风险。
- 保证任一阶段都能快速回退到 Python 基线。

## 切换原则
- 不一次性替换全部能力。
- 以“链路”为单位逐步切换，而不是按文件替换。
- 任何链路切换前，都必须完成 Python/TS 对照验证。

## 推荐切换顺序

### 第 1 批：设计稿单图与列表
- 切换 `lanhu_resolve_invite_link`
- 切换 `lanhu_list_designs`
- 切换单图 `detailDetach -> XDCover` 提取

切换条件：
- 能解析直接链接与邀请链接。
- 能下载正确的设计稿封面图。
- 与 Python 对照时，目标图名称、ID、URL 一致。

### 第 2 批：设计稿深度分析
- 切换 `lanhu_analyze_designs`
- 切换 `lanhu_get_slices`

切换条件：
- HTML 成功生成。
- `layout_summary` 存在。
- `design_tokens` 存在。
- schema 失败时 fallback 生效。

### 第 3 批：原型页链路
- 切换 `lanhu_list_pages`
- 切换 `lanhu_analyze_pages`

切换条件：
- 页面列表数量一致。
- 原型页文本提取结果可用。
- 样式摘要存在且字段完整。
- 不重新引入浏览器依赖。

### 第 4 批：外围与默认入口
- 切换 Docker、README、安装脚本、CI。
- 让 TS 版本成为默认服务入口。

## 上线方式建议

### 方案 A：双服务并行
- Python 服务继续跑在当前端口。
- TS 服务运行在新端口。
- 用同一批链接做对照验证。

优点：
- 回退成本最低。
- 便于逐条链路对照。

缺点：
- 运维复杂度稍高。

### 方案 B：单服务特性开关
- 统一入口。
- 按工具名或环境变量路由到 Python/TS 实现。

优点：
- 客户端配置不变。
- 易于逐工具灰度。

缺点：
- 服务内部复杂度更高。

## 推荐回退策略

### 服务级回退
- 保留 Python 入口和 Docker 镜像。
- 当 TS 任一关键工具异常时，直接切回 Python 服务。

### 工具级回退
- 如果采用单服务特性开关：
  - `LANHU_RUNTIME=python`
  - `LANHU_RUNTIME=ts`
  - 或更细粒度：
    - `LANHU_TOOL_ANALYZE_DESIGNS=python`
    - `LANHU_TOOL_ANALYZE_PAGES=python`

### 文档级回退
- README、部署文档切换前先保留 Python 启动方法。
- 只有当 TS 入口稳定后，再移除 Python 的主文档入口说明。

## 风险清单

| 风险 | 表现 | 应对 |
|---|---|---|
| 蓝湖接口字段变化 | TS 类型守卫失败或字段缺失 | 保留原始响应快照，异常时快速对照 Python |
| 单文件逻辑拆分后漏兼容 | 某些边缘链接或字段不工作 | 先写兼容矩阵，再逐项打桩验证 |
| HTML 转换不一致 | CSS 数值缺失、图片路径错误 | 对照 Python 输出的关键片段与映射表 |
| 原型页文本提取缩水 | 页面文本明显减少 | 用固定 HTML 样本做对照测试 |
| 脚本与 Docker 不一致 | 本地能跑，容器不能跑 | 每阶段同步维护 README 与容器启动方式 |

## 发布检查表
- TS 服务能启动并注册全部 MCP 工具。
- 设计稿单图提取通过。
- `lanhu_analyze_designs` 通过。
- `lanhu_analyze_pages` 通过。
- Docker 可运行。
- README 与实际启动命令一致。
- Python 回退入口仍可用。

## 回退触发条件
- 任一 P1 工具不可用。
- 设计稿 HTML 生成失败率明显上升。
- 原型页分析缺失文本或样式摘要。
- 线上链接解析成功率低于 Python 基线。

## 回退步骤
1. 停止 TS 默认入口或关闭 TS 特性开关。
2. 恢复 Python 服务为默认入口。
3. 复用兼容矩阵定位是接口层、转换层还是包装层问题。
4. 用固定样本补充测试后再重新灰度。
